# 解决方案模板

根据错误类型匹配预定义的解决方案。

---

## SQL 错误解决方案

### SQL语法错误修复

**匹配关键字**: `syntax error`, `unexpected token`

| 项目 | 内容 |
|------|------|
| 根因 | SQL语句存在语法错误，无法解析执行 |
| 步骤 | 1. 检查错误日志中指出的具体位置<br>2. 确认SQL关键字拼写正确<br>3. 检查括号、引号是否匹配<br>4. 确认语句符合当前引擎的SQL方言 |
| 预期效果 | SQL语句能够正确解析执行 |
| 注意事项 | 参考Lakehouse SQL语法文档；注意不同引擎的语法差异 |

---

### SQL对象不存在错误修复

**匹配关键字**: `table not found`, `column not found`

| 项目 | 内容 |
|------|------|
| 根因 | SQL中引用的表或列不存在 |
| 步骤 | 1. 确认表名/列名拼写正确<br>2. 检查是否指定了正确的schema/database<br>3. 确认表已创建且有访问权限<br>4. 如果是视图，确认视图定义正确 |
| 预期效果 | SQL能够正确找到引用的对象 |
| 注意事项 | 注意大小写敏感性 |

---

### SQL资源不足错误处理

**匹配关键字**: `out of memory`, `partition too large`

| 项目 | 内容 |
|------|------|
| 根因 | SQL执行过程中资源（内存/磁盘）不足 |
| 步骤 | 1. 分析job history，定位资源瓶颈阶段<br>2. 检查是否存在数据倾斜<br>3. 考虑增加分区或调整并行度<br>4. 优化SQL，减少中间结果集大小<br>5. 如需要，申请更多计算资源 |
| 预期效果 | SQL能够在资源限制内完成执行 |
| 注意事项 | 关注stage信息中的数据膨胀点；考虑分批处理大数据量 |

---

### SQL权限错误处理

**匹配关键字**: `permission denied`, `access denied`

| 项目 | 内容 |
|------|------|
| 根因 | 当前用户没有执行该SQL的权限 |
| 步骤 | 1. 确认当前执行用户<br>2. 检查用户对相关表/库的权限<br>3. 联系管理员授予必要权限<br>4. 或使用有权限的账号执行 |
| 预期效果 | 用户获得必要权限，SQL正常执行 |

---

## 调度错误解决方案

### 上游依赖失败处理

**匹配关键字**: `upstream failed`, `dependency failed`

| 项目 | 内容 |
|------|------|
| 根因 | 上游任务执行失败，导致当前任务无法开始 |
| 步骤 | 1. 调用get_task_run_dependencies获取依赖关系<br>2. 找到失败的上游任务<br>3. 分析上游任务失败原因并修复<br>4. 重新执行上游任务<br>5. 上游成功后重新调度当前任务 |
| 预期效果 | 依赖链路正常执行 |

---

### 上游任务超时处理

**匹配关键字**: `waiting for upstream`, `upstream timeout`

| 项目 | 内容 |
|------|------|
| 根因 | 上游任务执行时间过长，超过等待阈值 |
| 步骤 | 1. 检查上游任务当前状态<br>2. 如果上游仍在运行，评估是否需要等待<br>3. 如果上游卡住，考虑终止并重试<br>4. 调整依赖超时配置（如需要） |
| 预期效果 | 依赖等待正常完成 |

---

### 任务超时处理

**匹配关键字**: `timeout exceeded`, `kill`

| 项目 | 内容 |
|------|------|
| 根因 | 任务执行时间超过配置的超时阈值 |
| 步骤 | 1. 调用get_task_configuration_detail检查超时配置<br>2. 分析任务为什么执行时间长<br>3. 优化任务逻辑减少执行时间<br>4. 或适当增加超时阈值 |
| 预期效果 | 任务在超时时间内完成 |
| 注意事项 | 不建议无限制增加超时时间；应优先优化任务本身 |

---

### 调度资源不足处理

**匹配关键字**: `vc unavailable`, `queue full`

| 项目 | 内容 |
|------|------|
| 根因 | 计算资源（VC/队列）不可用或已满 |
| 步骤 | 1. 检查当前VC状态<br>2. 查看队列排队情况<br>3. 等待资源释放后重试<br>4. 或调整任务优先级<br>5. 如持续不足，申请扩容 |
| 预期效果 | 任务获得资源正常执行 |

---

## 数据集成错误解决方案

### 多表实时同步错误处理

**适用场景**: 实时同步任务（CDC、多表实时同步）

| 项目 | 内容 |
|------|------|
| 根因 | 实时同步任务执行异常 |
| 步骤 | 参考标准操作手册进行排查 |
| 参考文档 | https://yunqi.tech/documents/multitable_realtime_sync_sop |

---

### 离线批量同步错误处理

**适用场景**: 离线同步任务（批量同步、全量同步、增量同步）

| 项目 | 内容 |
|------|------|
| 根因 | 离线批量同步任务执行异常 |
| 步骤 | 参考标准操作手册进行排查 |
| 参考文档 | https://yunqi.tech/documents/batch_sync_Sop |

---

### 数据源连接失败处理

**匹配关键字**: `connection failed`, `source permission denied`

| 项目 | 内容 |
|------|------|
| 根因 | 无法连接到数据源或权限不足 |
| 步骤 | 1. 检查数据源连接配置（地址、端口）<br>2. 验证账号密码是否正确<br>3. 检查网络连通性<br>4. 确认源端账号有读取权限<br>5. 检查源端是否有IP白名单限制 |
| 预期效果 | 成功连接数据源 |

---

### 数据写入失败处理

**匹配关键字**: `write failed`, `target conflict`

| 项目 | 内容 |
|------|------|
| 根因 | 数据写入目标端失败 |
| 步骤 | 1. 检查目标端连接和权限<br>2. 确认目标表结构与源端匹配<br>3. 检查是否有主键冲突<br>4. 调整冲突处理策略（覆盖/跳过/报错） |
| 预期效果 | 数据成功写入目标端 |

## 数据质量错误解决方案

### 数据量异常处理

**匹配关键字**: `row count`, `empty result`

| 项目 | 内容 |
|------|------|
| 根因 | 数据量不符合质量规则预期 |
| 步骤 | 1. 检查DQC规则配置是否合理<br>2. 分析上游数据是否正常产出<br>3. 确认是否有数据过滤导致数量减少<br>4. 调整规则阈值（如确认数据正常） |
| 预期效果 | 数据量符合预期或规则调整合理 |

---

### 数据内容异常处理

**匹配关键字**: `duplicate`, `null value`

| 项目 | 内容 |
|------|------|
| 根因 | 数据内容不符合质量规则 |
| 步骤 | 1. 查看具体的校验失败记录<br>2. 分析异常数据的来源<br>3. 修复上游数据问题<br>4. 或调整DQC规则（如业务允许） |
| 预期效果 | 数据内容符合质量要求 |

---

## 通用兜底方案

当无法匹配到具体解决方案时，使用以下通用排查步骤：

1. 仔细阅读错误日志，理解错误含义
2. 检查任务配置是否正确
3. 查看历史运行记录，判断是否偶发
4. 搜索知识库或文档查找类似问题
5. 如无法解决，联系技术支持并提供完整日志

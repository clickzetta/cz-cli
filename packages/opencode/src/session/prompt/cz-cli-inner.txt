When the user asks operational questions about Clickzetta (e.g. listing workspaces, running SQL, checking task status), you MUST proactively use `cz-cli` via bash to take action, rather than asking the user for additional information.

You have `cz-cli` available in PATH for direct access to Clickzetta Studio and Lakehouse:
- Query workspaces, databases, schemas, tables and other metadata
- Run SQL queries
- Create, manage, and schedule tasks
- Check task run status and logs

## Output Formats

- `--format json` (default) — single-line JSON, best for parsing
- `--format toon` — line-per-field, good for `grep`/`head`
- `--format table` / `--format csv` / `--format pretty` — human-readable
- `--field <name>` — extract one field as raw text

Run `cz-cli <command> --help` for detailed options on any command.

## Key Rules

**SQL is async by default**. Use `--sync` for SELECT when you need data immediately.
**Write operations ALWAYS require `--write` flag** — this includes ALL DDL (CREATE, ALTER, DROP, CREATE DYNAMIC TABLE, CREATE MATERIALIZED VIEW) and ALL DML (INSERT, UPDATE, DELETE, MERGE INTO, COPY INTO). Without `--write`, the command will be rejected. Always add it proactively.
**Always pass `--type` when creating tasks** (SQL/PYTHON/SHELL/SPARK/FLOW).
**Flow tasks use `task flow *` commands exclusively** — never use `task save-content` or `task deploy` on flow nodes.
**Paginated results**: `list` commands return page 1 only. Check `ai_message` in response for next-page hints.
**State-changing operations** (deploy/undeploy/execute/delete/refill): confirm intent with user first.
**Multi-environment**: use `--profile <name>` to target a specific environment.
**On `NO_PROFILE` error**: guide user to run `cz-cli setup`.
**补数/回填/重跑历史数据 → `runs refill`**: when user says "补数", "回填", "重跑历史", "backfill", "re-run historical data", use `cz-cli runs refill <task> --from YYYY-MM-DD --to YYYY-MM-DD`. This is under `runs`, NOT `task`.
**Use a file for SQL with special characters**: if the SQL contains quotes, `$`, backticks, `\`, or newlines, run `cz-cli sql -f <file>` instead of inline `cz-cli sql "..."` (the shell corrupts inline SQL).
**Lakehouse SQL dialect ONLY**: cz-cli connects to ClickZetta Lakehouse, NOT Snowflake or Databricks. Do NOT use Snowflake syntax (e.g. `SHOW STAGES IN SCHEMA xxx`). Lakehouse uses `SHOW VOLUMES IN xxx` (no `SCHEMA` keyword). When unsure about syntax, consult the **lakehouse-doc** skill BEFORE running SQL — guessing Snowflake/Databricks syntax will fail.
**Break out of loops immediately**: If the same command fails 2+ times, or you notice you are repeating similar actions without progress, STOP. Do NOT retry with minor variations. Instead: explain what went wrong, state what you tried, and ask the user for guidance or try a fundamentally different approach.
**Never fabricate URLs or identifiers**: Surface links and IDs EXACTLY as they appear in command output. Do NOT hand-edit, truncate, or extrapolate them. If a command returned no value, do not invent one.

## Lakehouse SQL: Always Read the Docs First

Before generating, modifying, validating, explaining, or running any Lakehouse SQL — including:
- Writing new SQL queries or DDL/DML statements
- Querying system tables or metadata (e.g. `information_schema.*`, `SHOW ...`, `DESCRIBE ...`)
- Any task that touches ClickZetta Lakehouse SQL syntax

You MUST first load **lakehouse-doc-en** to verify the correct ClickZetta Lakehouse dialect. This is NOT Snowflake or Databricks syntax. Guessing syntax without consulting the docs will produce incorrect SQL.

This rule applies even for seemingly simple queries like `SELECT * FROM information_schema.tables` — the schema names, column names, and supported predicates differ from standard SQL and other platforms.

## Working style
- Build context from the codebase before acting. Do not guess when the repo can answer the question.
- Keep responses concise and factual. Do not use emojis unless the user explicitly asks for them.
- Do not begin responses with filler like "Got it" or "Sure".

- Never invent URLs. Only use URLs from the user, local files, or tools when you are confident they are relevant.

## Tool usage
- Prefer specialized tools over shell for file operations.
- Use `Glob` and `Grep` to find files and text.
- Use `Read` before editing an existing file.
- Use `Edit` for targeted changes to one file and `MultiEdit` when making several edits in the same file.
- Use `Write` only when you need to create or fully replace a file.
- Use `Bash` for terminal operations such as `git`, `bun`, builds, tests, and other system commands.
- Use `TodoWrite` for non-trivial work so progress is tracked clearly.
- Use `Task` for slash-command execution or well-scoped delegated work. Do not use it when a direct `Read`, `Glob`, or `Grep` lookup is faster.
- When multiple tool calls are independent, make them in parallel in a single response.
- Do not use shell commands like `cat`, `head`, `tail`, `sed`, `awk`, or `echo` when a dedicated tool is available.

## Editing approach
- Prefer the smallest correct change.
- Follow existing patterns before introducing new structure.
- Keep things in one function unless extraction clearly improves reuse or readability.
- Default to ASCII when editing or creating files unless the file already relies on Unicode.
- Add comments only when they help explain a non-obvious block.
- Avoid creating new files unless they are clearly necessary.

## Workspace hygiene
- The worktree may already contain user changes. Never revert or overwrite changes you did not make unless the user explicitly asks.
- If unrelated files are dirty, ignore them.
- If the files you need are already dirty, read carefully and adapt instead of discarding existing work.
- Avoid destructive git commands like `git reset --hard` or `git checkout --` unless the user explicitly asks.
- Do not amend commits unless the user explicitly asks.

## Execution

- Ask a question only when you are blocked by ambiguity, risk, or missing secrets that cannot be inferred safely.

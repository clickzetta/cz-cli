# 全增量一体同步功能

全增量一体同步功能是一项强大的数据同步工具，它能够将数据库中的数据完整地同步到Lakehouse中。该功能包括历史数据的全量同步和实时变更数据的增量同步。请注意，此功能目前处于 Beta 版本，部分已知局限将在后续版本中得到改进和优化。

## 任务创建流程

### 操作入口

在任务开发界面，点击新建操作按钮，从下拉菜单中选择“全增量一体同步”任务类型。

![新建全增量一体同步任务](.topwrite/assets/image_1710210458365.png)

### 配置来源数据类型

在创建任务时，首先需要选择一种需要同步的来源数据类型。目前支持的来源数据源类型及对应的增量读取模式和数据库版本如下：

| **类型**     | **增量读取模式** | **数据库版本** |
| ---------- | ---------- | --------- |
| MySQL      | Binlog     | 5.x、8.x   |
| PostgreSQL | WAL 日志     | 14 及以上     |

### 选择同步类型

全增量一体同步任务支持两种同步类型，请根据实际需求进行选择：

1. 实时同步-多表镜像：适用于将源端数据完整地镜像同步到目标端的场景。
2. 实时同步-多表合并：适用于将分库分表的数据合并后写入同一张目标表的场景。

#### 多表镜像

##### 来源配置
* 选择数据源：选择一个已有的数据源。如果没有可用的数据源，请参考[数据源配置](config-datasource.md)文档新建一个数据源。
* 配置读取模式：根据所选数据源类型，选择合适的读取模式。例如，MySQL数据源当前只支持BINLOG模式，而PostgreSQL数据源只支持WALs模式。
* 选择同步对象：选择需要同步的数据库和表。可以通过批量配置功能，上传配置文件来快速选择多个同步对象。请注意，配置文件中的注释说明需要删除。
* 配置槽位（slot）
  * 仅PostgreSQL需要配置，槽位（slot）指PostgreSQL 的 replication slot，详见 [说明文档](https://www.postgresql.org/docs/9.4/catalog-pg-replication-slots.html)。
  * 每个数据库都需要配置一个槽位。目前支持decoderbufs和pgoutput两种插件类型。
  * 可以选择已存在的槽位进行使用；或者在页面页面来操作新建，在新建窗口的下方区域的创建语句，可按需修改。点击确定后进行创建。在数据源中配置的用户名必须具备创建slot的权限，否则会创建失败，详见下文的权限说明。
  * 特别注意，不同任务不要复用同一个槽位。任务启动时，如果槽位被其他正在运行的任务占用，会启动失败。如果其他任务处于停止状态，因槽位中的数据会被共享消费，位点是共享的，新任务消费的增量数据，同步到目标端的表数据可能会不全。

##### **目标配置**
* 目标数据源：默认选择当前工作空间对应的Lakehouse数据源。
* 命名空间规则：目前仅支持选择特定的命名空间。未来版本将支持按照源端进行镜像命名或自定义命名规则。
* 目标表命名规则：目前仅支持镜像来源表名。后续版本将支持添加前缀或自定义命名规则。
* 计算集群：选择工作空间下可用的计算集群。建议使用AP类型的集群。

##### 预览配置
在完成源端和目标端配置后，可以预览同步表和字段的映射关系。如有需要修改，可以返回上一步进行调整。

##### 同步规则
在同步规则中，可以配置Schema Evolution等规则，定义源端表和字段变化后的处理行为。

#### 多表合并

##### **来源配置**

* 选择数据源
  * 同上，多表镜像同步类似
* 配置读取模式
  * 同上，多表镜像同步类似
* 选择同步对象
  多表合并同步，会使用“虚拟表”作为中间的过渡承接，需要在配置时指定使用什么虚拟表来承接哪些对象的数据。支持两种方式进行配置
  * 方式一：基于规则配置，通过正则匹配进行筛选，比如选出以 abc 开头的所有表，在正则表达式中填写 ^abc 即可
  * 方式二：基于文件批量配置，按照模板中格式要求填写配置文件、上传后，页面会按照文件中给定的内容来选中需要同步的对象，选中后可以按需再进行微调。请注意删除模板文件中的注释说明

##### **目标配置**
* 同上，和多表镜像同步类似

##### **预览配置**
* 同上，和多表镜像同步类似

##### **同步规则**
* 同上，和多表镜像类似

### 配置高级参数
在任务常规配置完成后，点击“配置”按钮完成高级参数的设置。
* 集群：指定用于运行同步任务的集群。建议使用AP类型的集群。
* 更新频率：设定数据写入的频次间隔。

### 提交任务
点击提交按钮，将任务提交到生产环境执行。请注意，任务提交后默认不启动，需要手动启动。

## 任务运维

### 启动任务

在任务详情页，可以操作启动任务。以下是三种启动方式：
1. 无状态启动（仅适用于任务首次启动）：完整同步所有数据，先进行全量同步，再启动增量同步。
2. 从上次保存状态恢复（适用于停止后的任务再次启动）：从停止位置断点续传。
3. 自定义起始位置：从给定的位点进行同步，可用于数据回刷。对任务中配置的所有表统一生效。

* MySQL：选择指定文件启动或者指定时间启动。在页面监控区域可以获取到上次文件位置。
* PostgreSQL：指定 LSN 取值，在下文的实例监控中可以查看到最新的 LSN 取值。

### 实例监控

任务启动后，会经历初始化、全量同步、增量同步三个阶段。在实例监控区域，可以查看这三个阶段的运行状态。

### 指标监控

在指标监控区域，可以查看全量同步和增量同步的关键监控指标。
![指标监控](.topwrite/assets/image_1705386727702.png)

| **指标名称**   | **口径说明**                                               |
| ---------- | ------------------------------------------------------ |
| 读取数据       | 数据同步任务在统计周期内从数据源读取的记录数。                                |
| 写入数据       | 数据同步任务在统计周期内向目标数据源写入的记录数。                              |
| 平均读取速率     | 数据同步任务在统计周期内平均读取速率。（周期内总读取记录 / 周期时间）                     |
| 平均写入速率     | 数据同步任务在统计周期内平均写入速率。（周期内总写入记录 / 周期时间）                     |
| Failover 次数 | 数据同步任务在统计周期内运行发生 Failover 次数。Failover 次数代表数据同步服务自身运行的稳定性。 |

### 同步对象

在同步对象区域，可以查看单表的同步终态和进行对应的运维操作。
![](.topwrite/assets/image_1710214491047.png)

| **指标** | **口径说明**                                    |
| ------ | ------------------------------------------- |
| 最新读取位置 | 同步任务实时读取来源对象数据并写入目标表，以目标表最新一条记录的写入时间作为读取位置。 |
| 最新更新时间 | 最近一次写入目标表的时间。                               |
| 数据延迟   | 数据从数据源端事务提交到同步至目标端可见的时间间隔。                  |

| **操作** | **行为定义**                                     |
| ------ | -------------------------------------------- |
| 全量同步详情 |                                              |
| 暂停     | 对于正在同步的表可操作，暂停该表的同步，保存同步位点。                  |
| 恢复     | 从上次暂停的同步位点开始继续同步。                            |
| 重新同步   | 对该表重新进行全量和增量同步。                              |
| 查看异常   | 点击后可以查看表的增量同步任务的异常信息，比如 Schema Evolution 的异常等。 |

### 停止任务

停止任务会停止任务中正在进行的全量同步和增量同步。停止时会自动保存增量同步的位点状态。

* 如果停止时表处在全量同步阶段，重启后，未完全全量同步的表，会重新进行一次全新的全量同步。
* 如果停止时表处在增量同步阶段，重启后，默认会从同步停止的位点继续同步。

### 下线任务

下线任务属于高危操作，任务下线后，不会保存当前同步的位点。如果再次上线并启动任务，会重新开始同步数据。

* 下线不会清理已经同步到目标端的数据，但会清理中间过程的缓存数据和位点信息。
* 重新同步不会重新建表，全量数据同步会对于老的表进行覆盖写入，增量同步会对目标表做merge into的更新。

## 权限说明

同步不同类型数据源的变更事件时，需要在相应数据源服务端配置好适当的权限才能保证数据的正常同步。尽管直接分配管理员或超级用户的权限足以保证任务正常运行，但通常希望最小化需要分配给同步数据用户的权限。各操作步骤需要的具体权限配置说明如下。

### PostgreSQL

| 需求                                     | 权限                                                                    | 示例                                                                                                                                                                                   |
| -------------------------------------- | --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 配置任务（获取元数据：schema列表、table列表、table字段列表） | SELECT（information\_schema和需要获取详情的table上）                             | 赋予某个角色读取information\_schema的权限：GRANT SELECT ON TABLE information\_schema.tables TO role\_name;&#xA;&#xA;赋予某个角色读取某张表的权限：GRANT SELECT ON TABLE your\_schema.your\_table TO role\_name; |
| 同步wal日志                                | REPLICATION&#xA;LOGIN                                                 | CREATE ROLE \<name> REPLICATION LOGIN;                                                                                                                                               |
| 同步历史全量数据（可选）                           | SELECT（需要同步的table上）                                                   | 赋予某个角色读取某张表的权限：GRANT SELECT ON TABLE table\_name TO role\_name;&#xA;&#xA;赋予某个角色读取schema下所有表的权限：GRANT SELECT ON ALL TABLES IN SCHEMA schema\_name TO role\_name;                      |

## 已知局限

* 当前方案需要设定数据刷新频次，最少支持设置为 1 分钟。即端到端数据新鲜度延迟至少会在 1 分钟以上，尚未做到秒级延迟（会在后续版本提供）。
* Schema Evolution，暂不支持变更字段类型、不支持自动新增表。
* 多表实时同步任务中，源端不同库表中若存在主键相同的数据，同步结果会出现异常。
* MySQL同步，尚不支持的字段类型：

| 字段类型 | 同步后行为 |
| ---- | ----- |
| year | 取值不对应 |

* PostgreSQL同步，尚不支持的字段类型：

| 字段类型     | 同步后行为          |
| -------- | -------------- |
| varbit   | 取值不对应          |
| bytea    | 取值不对应          |
| TIMETZ   | 取值不对应          |
| interval | 取值不对应          |
| NAME     | 取值不对应          |
| NUMERIC  | 精度不对应，目标端精度会更高 |
| decimal  | 精度不对应，目标端精度会更高 |

^

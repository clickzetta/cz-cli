# 补数任务

补数据功能可通过补充历史或未来一段时间的数据，将数据写入对应时间分区。若代码中配置了调度参数，系统将根据补数据选择的业务时间自动替换为具体值，并结合业务代码将对应时间的数据写入指定分区。具体写入的分区与执行的代码逻辑由任务定义中的代码决定。

## **功能使用场景**

* 当任务新建后，若希望对历史数据进行加工，可通过补数据功能选择历史某个时间段来执行任务；
* 当任务在某个时间段内数据有问题或者任务本身异常时，可通过补数据功能对历史异常的数据或任务重新生成实例并执行，以达到修复历史数据的目的。

## 功能操作

### 步骤一：新建补数据任务

1. 点击左侧导航**运维监控 -> 任务运维**，选中**周期任务** Tab，并选择其中任意一个已管理的周期任务，点击操作栏中的“补数据”，进入补数据操作界面。
2. 配置补数任务的相应参数

* 任务名称：系统默认名称，暂不支持用户修改

* 选择补数任务包含的任务节点

  * 包含当前节点：用户可根据实际需求选择是否勾选当前任务参与补数。
  * 包含下游节点：用户可根据实际需求勾选需要补数据的**下游任务链路**，支持不选择、选择全部、自定义选择范围三种模式。

* 任务定时时间范围：在创建补数据任务时，支持添加多段计划执行时间范围，用于满足补充多段不连续时间的需求。目前最多支持添加4段计划执行时间，且不允许选择重叠的时间段。

* 最大并发数：表示当连续补充多个周期的数据时，若同时运行的实例数达到此并发上限，则可以并行运行多个周期的实例，从而提升补数据的效率，但也会消耗更多的资源。

  * 最大并发数量，指的是同时运行的最大实例个数。

* 补数自依赖：若用户需要补数任务严格按照业务时间依次往后执行，可选择补数自依赖选项。

  * 若用户同时配置了最大并发数和补数自依赖，则整体执行效率可能会下降。

^

最大并发数与补数自依赖对执行效率的影响：

* 配置最大并发数时，若该周期任务本身没有配置自依赖，同时也没有勾选补数自依赖选项，系统会基于用户配置的并发数，按照系统资源、依赖关系等随机分配执行的具体实例。
* 若周期任务本身配置了自依赖，或勾选了补数自依赖选项，则同时并发的实例数会受到实例自身顺序依赖逻辑的影响，再结合系统资源、依赖关系等随机分配执行的具体实例，效率会降低。

### 步骤二：查看补数据任务

1. 点击左侧导航**运维监控 -> 任务运维**，选中**补数任务** Tab。
2. 基础信息：包括并发分组、目标任务ID、任务名称、工作空间、是否包含下游、状态、执行顺序、任务定时时间范围、提交时间、提交人等。
3. 统计图：

* 全局统计信息：可查看当前补数任务内包含的所有计划时间点总数，并按天粒度展示执行状态。

* 按粒度统计图：提供年、月、日三种粒度下的数据统计，点击统计图会联动下方的实例列表。

  * **年视图**：鼠标悬停在日期上方可查看当天的执行情况；
    ![](.topwrite/assets/image_1723185910796.png)
  * **月视图**：可在月视图上查看当天要执行的实例总数，鼠标悬停在日历上方可查看执行状态汇总；
    ![](.topwrite/assets/image_1723185934736.png)
  * 日视图：支持按天或按小时两种粒度切换，可查看不同聚合粒度下实例的平均耗时时长。
    ![](.topwrite/assets/image_1723185952493.png)

4. 实例统计列表：默认按照计划时间展示当前补数任务下所有执行实例的信息，点击可跳转到实例执行的详情页面查看依赖关系等详细信息。

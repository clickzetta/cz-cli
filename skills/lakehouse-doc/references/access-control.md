# 授权

## 1. 访问控制模型

云器Lakehouse的所有元数据对象均基于访问控制体系进行授权访问。云器Lakehouse支持的访问控制模型包括**访问控制列表**（ACL，Access Control List）和 **基于角色的访问控制**（RBAC，Role-Based Access Control）两种。

* **访问控制列表**（**ACL，Access Control List**）

权限点被直接授予用户（User）。

* **基于角色的访问控制**（RBAC，Role-Based Access Control）

权限点被授予角色（Role），用户通过被授予角色而获得相应的权限。

* **DAC（Discretionary Access Control）模式兼容**

云器Lakehouse不直接支持DAC模式。DAC模式指：元数据对象（table、view、virtual cluster等）的所有者（owner）默认拥有该元数据对象的所有权限，且可将权限授予其他主体。云器Lakehouse中的元数据对象被创建后，其创建者默认具备该对象的所有权限，但无法进行二次授权。

^

云器Lakehouse访问控制体系中的主要概念包括：

* **元数据对象（Object）**

元数据对象是一个可以被授出访问权限的实体。一个元数据对象包含多个权限点，除非通过明确授予允许，否则相关操作将被拒绝。

* **角色（Role）**

角色是一个可以被授予权限的实体。角色可以分配给用户，但不可以分配给其他角色。

* **权限（Privilege）**

权限是对元数据对象访问的最小定义级别。可以使用多个不同的权限点来控制授予操作的范围。

* **用户（User）**

用户是云器 Lakehouse 识别的用户身份，可以是用户身份或服务使用的身份。

## 2. 角色

### 2.1 角色类型

云器Lakehouse中的角色两种：**预置角色**和**自定义角色**。

为便于用户快速完成基本授权并使用产品功能，云器Lakehouse预置了多个角色。预置角色随服务实例创建而自动创建，不可修改权限或删除，但可以被授予用户。

用户可根据自身需要，创建自定义角色，并对该角色授权。自定义角色可随时修改角色的属性、权限范围或删除。目前仅限在工作空间内创建自定义角色，不支持创建实例级别的自定义角色。

### 2.2 角色级别

云器Lakehouse的元数据对象分为“服务实例”（Instance）和“工作空间”（workspace）两个级别，角色也对应分为“实例角色”和“空间角色”两个级别。两个级别的对象没有从属关系，实例级别的角色不具备且不可被授予工作空间级别任何元数据对象（空间用户、空间角色、schema、table、计算集群等）的任何权限。

空间角色以工作空间（workspace）为隔离边界。不同工作空间内，允许存在同名的角色。不同工作空间之间，不允许跨空间将权限授予另一工作空间角色。

### 2.4 用户自定义角色

当前仅支持在工作空间内，使用SQL语法创建自定义角色。创建自定义角色的语法为：

```Plain
CREATE ROLE <role_name>;
```

仅被授予工作空间的空间管理员角色（workspace\_admin）的用户，可在工作空间内创建自定义角色，并可随时对自定义角色进行权限管理和删除操作。

## 3. 权限（Privileges）

### 3.1 权限定义

访问控制权限的授予决定了用户可以对特定对象执行指定操作。对于云器Lakehouse中的每个元数据对象对象，都有一组可以授予的权限。云器Lakehouse中的所有元数据对象及其拥有的权限点，详见 权限点 文档。

元数据对象有三种表达方式：

1）以对象类型加对象名称，表达已被创建的对象。例如：table mytable（名称为mytable的表） 或 vcluster myvcluster（名称为myvcluster的计算集群）；

2）以关键字ALL 加对象类别，表达现有和未来所有该类型的对象。例如：ALL tables in schema my\_schema （名称为my\_schema的schema下的所有table类型对象）；

3）以关键字ALL 加关键字 OBJECTS 表达现有和未来的所有类型的所有对象，例如：ALL OBJECTS in workspace my\_workspace（名称为my\_workspace的工作空间下所有的对象）。

权限点有两种表达方式：

1）以具体权限点表达，例如：select，read metadata，update 等；

2）以关键字ALL加关键字PRIVILEGES，表达所有权限点，例如：ALL PRIVILEGES on table mytable（mytable表的所有权限点）。

在使用SQL语句进行权限管理时，使用 GRANT <权限点> 和 REVOKE <权限点> 命令。在GRANT <权限点>语句末尾加上with grant option，表示在授予该权限点的同时，将允许被授予者将该权限点再授予其他角色或用户。

业务对象不可通过SQL进行授权。用户通过被授予包含业务对象权限的预置角色来获取相关权限。暂不支持细粒度授权。

### 3.4 ALL 对象

ALL 对象是一种特殊的对象，表达一类对象下**当前**和**未来**的所有个体。例如：all tables表达当前和未来的所有table；all vclusters表达当前和未来的所有计算集群。

通过`ALL`关键字可以一次性为对象集合（如某schema下的所有表）授予权限，从而减少繁琐的重复授权操作。同时，ALL关键字提供“Future Grants”（未来授权）机制，例如：`all tables in schema s1` 表示schema s1内当前和未来可能新增的所有table类型对象。

ALL 对象在使用时，需要和关键字“in”一起使用，以明确ALL 对象描述的范围，关键字“in”之后跟随的是ALL 对象的父级对象。例如：all tables in schema my\_schema; all vclusters in workspace my\_workspace。

ALL 对象的一种特殊用法是用于指代某父级对象下的所有自级对象，例如：all objects in schema my\_schema，表达my\_schema下的所有类型对象，包括table、view、functions等。

注意：尽管ALL对象可以提升授权操作的效率，但请谨慎使用all objects的方式授权。仅在确认all objects指代的范围符合授权预期时，再使用all objects进行指代，避免授权范围超出预期。

### 3.6 ALL 权限点

ALL 权限点是各类元数据对象上均存在的一个权限点，表达该对象的所有权限点。ALL权限点是一个**独立**的权限点，与其他权限点并非指代关系。因此授予ALL权限点不会覆盖授予其他权限点，解除ALL权限点也不会解除其他权限点。

例如：对user my\_user 先后授予了example\_table的select和ALL权限点。则此时my\_user用户由于被授予了ALL权限点，可以对example\_table执行所有操作；在查询my\_user用户对example\_table的权限时，显示同时拥有的select和all权限点的授权。当对my\_user用户解除ALL权限点的授权时，my\_user用户仍具备对example\_table进行select操作的权限。

使用ALL权限点可以快速完成对一个对象所有权限的授予，例如：

```SQL
Grant all privileges on table example_table to user my_user;
```

## 4. 授权（Grants）

### 4.1 授权定义

授权的基本模型是将“**对象**”的“**权限点**”授予“**身份**”。其中对象可以是元数据对象或业务对象，权限点是这个对象包含的一个或多个权限点，身份目前指“用户（user）”和“角色（role）”两种。

### 4.2 授权关系管理

一个用户（user）可以通过两种方式获得某个对象的权限：1）直接被授予该对象的权限；2）通过被授予拥有该对象权限的角色获得。

当一个用户同时被授予多个角色时，该用户的权限范围是多个角色权限范围的并集。

角色不允许被授予给角色。

### 4.3 查看授权

支持以被授予身份和被授权对象两个角度来查看授权。

从被授权身份角度查看，例如：show grants to user my\_user; 将显示该用户被授予的所有角色，以及被**直接**授予的权限。

从被授权对象角度查看，例如：show grants on table my\_table; 将显示该对象所有被授出的权限，包含授予用户的和角色的。

### 4.4 授权操作

元数据对象的授权操作，可以使用SQL操作，详见GRANT和REVOKE语句的相关介绍。也可以在“管理”-“安全”-“权限”页面中进行Web端操作。

业务对象的授权目前仅支持通过授予预置角色的方式授予，暂不支持单独对业务对象授权。
